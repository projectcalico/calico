# Semaphore → ArgoCI e2e migration

Status: DRAFT v4 (built + validated on kind@v3.7.4; ops onboarding pending)
Owner: Lance · Reference pipeline: `.semaphore/end-to-end/pipelines/nftables.yml`

## Summary

Move OSS Calico's **scheduled** e2e suites off Semaphore onto ArgoCI (Argo
Workflows), for an internal goal of end of July 2026 (Semaphore the platform
stays; we just stop using it for OSS e2e). Each Semaphore pipeline becomes an
Argo `CronWorkflow` in the repo under `.argoci/`, generated by a Go converter
(`hack/cmd/semaphore2argo`) and verified by a job-parity harness
(`hack/cmd/argo-jobreport`). `nftables.yml` is the proving case; the converter
handles all pipelines × active branches (`master`, `release-v3.32`,
`release-v3.31`). Conversion is **faithful** — same jobs, same params, same
`bz` flow.

## Decisions that need careful review

Everything else in this doc is mechanism or settled fact; these three are the
non-obvious calls worth scrutiny:

1. **Per-branch checkout** (§Runtime). Each cron checks out its own branch, so
   the `.argoci/` scaffold must be **cherry-picked to every release branch**.
   Faithful to Semaphore, but 3 branches to keep in sync. Alternative
   (master-tooling + `RELEASE_STREAM` param, no cherry-picks) was rejected — is
   that the right call?
2. **Dual-mode test selection** (§Selection). The converter passes through
   whichever selection the source declares: `E2E_TEST_CONFIG` (structured
   config) or `K8S_E2E_FLAGS` (regex). This is forced by `E2E_TEST_CONFIG`
   being **master-only** — the `e2e/pkg/testconfig` mechanism does not exist on
   v3.31/v3.32. Review the pass-through + the master-only constraint.
3. **Teardown on cancel/timeout** (§Lifecycle). Each job destroys its own
   cluster in its epilogue; `onExit` adds a best-effort sweep. But a pod killed
   by `activeDeadlineSeconds` may leave an orphaned cloud cluster — the durable
   safety net is a standing **janitor cron that is not yet built**. Is
   best-effort + janitor acceptable?

## Non-goals

PR-triggered e2e (separate security review); changing test behaviour;
Enterprise/CC pipelines; replacing `bz`; backporting `testconfig` to release
branches (not needed — dual-mode; cheap to do later, see §Migration path).

## Approach

**Central insight.** Every e2e job has the identical lifecycle (prologue →
provision → install → configure → migrate → test → epilogue) across all
streams and every pipeline file. Only the **block/job/matrix/env tree** and the
**selection mechanism** differ. So we hand-author the lifecycle once (`.argoci/`
scaffold + shared `WorkflowTemplate`s) and mechanically generate the per-job
DAG per pipeline.

**Converter** (`hack/cmd/semaphore2argo`, lib `hack/argoci/convert`). Reads a
Semaphore pipeline; each job → one DAG task that `templateRef`s the shared
`e2e-test` template. A job matrix → `withItems` (cartesian product), matrix
axes emitted as `{{item.X}}`. Env is merged global ⊕ block ⊕ job ⊕ matrix and
passed as one `test-vars` blob (`export K='v'` lines, single-quoted so regex
`K8S_E2E_FLAGS` survives). Deterministic: no inference, source-order matrix,
slug+dedupe task names, `PROVISIONER=local-kind` → large pod profile else
small. Emits `# CONVERTER-TODO` + non-zero exit for anything non-mechanical.
CLI: `--in --branch --schedule [--name --out]`; `RELEASE_STREAM` derives from
the branch at runtime.

**§Runtime — per-branch checkout.** Each cron checks out its own target branch
and runs *that branch's* `.argoci/` tooling + e2e tree; `RELEASE_STREAM`
derives from the branch. Consequence: the scaffold is authored on `master` and
cherry-picked to the release branches. The shared `e2e-test` /
`e2e-sweep-destroy` `WorkflowTemplate`s carry no branch-specific logic (they run
the checked-out branch's scripts by path), so they are applied to the
namespace once and every branch's crons `templateRef` them.

**Namespace (decided): `argoci-oss-e2es`.** OSS Calico e2e runs in its own
namespace, not the shared `argoci` where the private CC/tesla workloads live —
Calico is a public repo, so its workflows and ServiceAccount must not sit
alongside the private CC secret set. Only the OSS secrets live there.

**§Selection — dual-mode.** `E2E_TEST_CONFIG` (structured `e2e/config/*.yaml`:
`extends`/`include`/`exclude.labels`/`exclude.namePatterns`) is the new,
nicer mechanism but exists only on master; release branches use
`K8S_E2E_FLAGS` regex. The converter emits whichever the source job declares,
so master can adopt `E2E_TEST_CONFIG` incrementally with no converter change.

**§Lifecycle.** Per task: `global_prologue.sh` (setup, secrets via
`createLocalSecret`, `RELEASE_STREAM`/`CLUSTER_NAME`) → `body_standard.sh`
(phase dispatch) → `global_epilogue.sh` (diags + GCS + Lens + `bz destroy`).
Workflow `onExit` → best-effort sweep. Durable orphan cleanup = a standing
janitor cron (**to build**).

## Test strategy: job-parity harness

Faithful = the generated crons enumerate the same jobs+params as the Semaphore
pipeline. `hack/cmd/argo-jobreport` re-enumerates a generated cron by parsing
the emitted YAML **independently of the converter** (expands `withItems`,
parses `test-vars`) and `--diff`s it against the Semaphore side (the existing
`.semaphore/end-to-end/report/generate_e2e_report.py`, reused as the trusted
source of truth). Comparison is on significant params (infra vars ignored, per
that tool's list). Wired as the converter's CI gate; today: `PARITY OK,
10 jobs`.

## Findings (settled, from local kind@v3.7.4 validation)

The generated `e2e-nftables-master.yaml` lints and runs green end-to-end on a
local kind + Argo v3.7.4 (matching prod) with the real templates + real
prologue/epilogue and `body_standard`/`bz` stubbed: 10 pods via `withItems`,
matrix substitution + `K8S_E2E_FLAGS` escaping intact, `E2E_TEST_CONFIG`
passes through, `onExit` runs. Along the way:
- Argo requires `metrics.gauge.realtime` (added; valid on v3.7.4 too).
- **`templateDefaults` does not cross `templateRef`** → the shared templates
  pin their own `image`/`command`/`envFrom`; the cron carries only
  workflow-level `nodeSelector`/`tolerations` + the DAG.

## Migration path to full E2E_TEST_CONFIG (later)

Dual-mode gets us off Semaphore now. To unify on `E2E_TEST_CONFIG` later:
backport `e2e/pkg/testconfig` + `e2e/config/` to a release branch, author its
config files, then re-run the converter (a pure function of the source) to
regenerate that branch's crons. Low lock-in: the only throwaway is the Mode-2
code path. Backporting changes the release-stream e2e *binary* (CI tooling, not
a shipped feature) — a lighter release-policy call, but still the owners'.
